# 0415 작업 기록

## 목표
- HOLMES full(`5,000,000` events) 런을 `OOM/SIGKILL` 없이 완주시키기
- 후반부 속도 급락 원인 분리
- 분석 체계를 크게 망가뜨리지 않는 선에서 메모리 방어 장치 추가

## 핵심 진단 흐름
1. 초반 병목은 `ac_min`/ancestor cache 쪽이었다.
2. 그다음 병목은 `online_full_resync`였다.
3. 그다음 병목은 Rust native shadow가 read-primary 해제 후에도 계속 쌓이는 문제였다.
4. 현재 남은 핵심 병목은 `raw provenance graph` 자체와, 이를 막기 위해 너무 자주 도는 `graph GC`다.

## 실험 중 확인된 사실

### 해결된 것
- `ancestor/min_dist` 캐시 폭발
  - `graph_on_demand_ancestor_enabled=1`
  - `graph_ancestor_cache_entry_count=0`
  - `graph_min_dist_entry_count=0`
- `online_full_resync` 폭발
  - local eviction 패치 이후 `online_full_resync_time_seconds=0.0` 확인
- native shadow 재폭발
  - native backend auto-disable 이후
  - `native_backend_enabled=0`
  - `native_graph_node_count=0`
  - `native_graph_edge_count=0`
- `events_by_id` 무한 증가
  - `HOLMES_EVENT_META_SOFT_LIMIT`로 `retained_event_meta_count`를 `500000` 수준으로 유지

### 아직 남은 문제
1. 메인 파이프라인은 여전히 `1코어` 중심
   - Python mutable state(`graph`, `online_index`, `builder`, trackers`)가 메인 병목
2. raw graph 본체가 큼
   - hard-cap과 stale taint eviction을 넣은 뒤에도 raw graph 관리 비용이 큼
3. `graph_gc_every_events=5000`일 때 GC churn이 과함
   - 메모리는 안정되지만 속도가 심하게 저하됨

## 최근 런 요약

### `stage2_full_local_evict_v3`
- 개선점:
  - native shadow 꺼짐
  - full resync 없음
- 결과:
  - 약 `83%` 부근에서 `Killed`
- 해석:
  - 당시엔 raw graph 자체가 계속 커지는 게 주 원인으로 보였음

### `stage2_full_local_evict_v6`
- 설정:
  - hard-cap prune + stale taint eviction + event meta soft limit + native auto-disable
- 현재 확인 수치(마지막 체크 시점):
  - `events_processed = 2,935,193 / 5,000,000` (`58.70%`)
  - `events_per_second = 203.5`
  - `RSS ≈ 25.2 GB`
  - `native_backend_enabled = 0`
  - `online_full_resync_time_seconds = 0.0`
  - `graph_version_node_count = 2,505,608`
  - `graph_edge_count = 3,842,592`
  - `retained_event_meta_count = 500000`
  - `pruned_event_meta_count = 2,354,830`
- 해석:
  - 메모리 방어는 성공
  - 대신 `graph_gc_every_events=5000` 때문에 `GC churn`이 심해짐
  - 살아는 있으나 실질 속도는 너무 낮음

## 현재 판단
- 지금 가장 큰 문제는 `OOM`보다 `GC cadence`다.
- 방어 장치는 이미 충분히 들어갔다.
- 지금은 `GC 강도`를 줄일 게 아니라 `GC 빈도`를 낮춰야 한다.
